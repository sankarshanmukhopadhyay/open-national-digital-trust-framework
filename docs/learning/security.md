# Security Path

**Audience:** Security architects, threat modellers, operational-risk teams, and reviewers  
**Prerequisites:** [ONDTF in One Hour](one-hour.md), or equivalent familiarity  
**Estimated time:** 3–4 hours  
**Outcome:** a role-specific understanding of the framework, with enough context to use the reference documentation confidently.

## Recommended sequence

| Order | Read | Why it comes here |
|---:|---|---|
| 1 | [Security architecture](../security/security-architecture.md) | Understand the complete security posture. |
| 2 | [Security principles](../security/security-principles.md) | Establish design constraints. |
| 3 | [Security objectives](../security/security-objectives.md) | Identify intended security outcomes. |
| 4 | [Protected assets](../security/protected-assets.md) | Understand what requires protection. |
| 5 | [Trust assumptions](../security/trust-assumptions.md) | Expose assumptions that can fail. |
| 6 | [Security boundaries](../security/security-boundaries.md) | Locate consequential crossings. |
| 7 | [Adversary model](../security/adversary-model.md) | Understand positions and capabilities. |
| 8 | [Attack surfaces](../security/attack-surfaces.md) | Identify exposed institutional and technical surfaces. |
| 9 | [Threat catalogue](../security/threat-catalogue.md) | Review stable threat records. |
| 10 | [Control framework](../security/control-framework.md) | Map treatment to evidence. |
| 11 | [Threat-control scenario trace](../reference-scenario/threat-control-trace.md) | See the model instantiated. |

## Reading method

Read the pages in order on the first pass. On later visits, use the sidebar and search as a reference system. The sequence is curated; it does not imply that later pages are normatively subordinate to earlier ones.
