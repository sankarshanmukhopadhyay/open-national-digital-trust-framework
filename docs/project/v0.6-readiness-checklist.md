# v0.6.0 readiness checklist

## Release claim

v0.6.0 is the **Operational Framework Draft**. The claim is that ONDTF now supplies an internally integrated normative, institutional, lifecycle, assurance, conformance, rights, profile-construction and maintenance architecture, demonstrated through a worked generated profile. It is not a claim of production certification or independently proven interoperability.

## Operational architecture

- [x] Role-specific normative obligations are published and traceable.
- [x] Institutional decision rights and separation-of-duty expectations are published.
- [x] Provider and service lifecycle states and governed transitions are machine-readable.
- [x] Assessment, accreditation, certification, surveillance, suspension, withdrawal and appeal are specified.
- [x] Assurance dimensions and critical non-compensable floors are defined.
- [x] Affected-party notice, challenge, correction, appeal, accessibility and remedy are operationalised.
- [x] Profile types, composition, validation, precedence, dependency and change rules are defined.
- [x] Controlled-document, reference and profile maintenance procedures are established.
- [x] External-framework pattern adoption has a formal decision and traceability process.

## Guided construction and evidence

- [x] Guided construction stages, questions, patterns, contradiction rules and completeness gates are machine-readable.
- [x] A structured response schema distinguishes inherited, selected, provisional, unresolved and non-applicable decisions.
- [x] A complete fictional worked operational profile is generated and validated.
- [x] The worked profile preserves an explicit legal-authority deployment qualification.
- [x] Documentation architecture and guided-learning paths connect construction decisions to source obligations and generated artefacts.

## Issue disposition

- [x] URI-07, URI-08, URI-09, URI-10, URI-11, URI-12, URI-18 and URI-19 are closed with recorded evidence.
- [x] URI-06 and URI-17 are narrowed to pilot-validation evidence rather than missing operating architecture.
- [x] Remaining implementation, interoperability, calibration, accessibility, resilience and recognition issues are disclosed and assigned to later release gates.
- [x] Known limitations distinguish enduring boundaries from reducible maturity constraints.

## Publication and quality gates

- [x] Version-bearing repository files identify v0.6.0 and Operational Framework Draft.
- [x] Release notes, manifest, inventory, checklist and machine-readable release record are present.
- [x] YAML and JSON artefacts parse and repository validators pass.
- [x] Markdown publication coverage, local links and navigation integrity are validated.
- [x] Mermaid source validation passes.
- [ ] Confirm the release commit's GitHub Pages deployment and rendered Mermaid checks in GitHub Actions.
- [ ] Create and verify the `v0.6.0` tag and GitHub release after merge.

## Exit decision

The repository payload is ready for release when all local candidate checks pass. The two unchecked items are post-merge platform actions and do not represent missing specification content. A failed Pages build, broken rendered navigation or failed Mermaid rendering must block publication of the GitHub release until corrected.
